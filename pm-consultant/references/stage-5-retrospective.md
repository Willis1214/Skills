# Stage 5: Project Retrospective

## Trigger

Use only when the user explicitly asks for a retrospective report, project evolution summary, full process summary, or methodology distillation.

## Goal

Objectively reconstruct the project's thinking path, decision changes, technical tradeoffs, final assets, unresolved issues, and next-step recommendations.

## Rules

- Separate confirmed history from inference.
- Do not invent completion evidence, test results, or asset paths.
- If project records are incomplete, mark gaps as `TBD` or `未确认`.
- Keep the summary decision-oriented and useful for future reuse.
