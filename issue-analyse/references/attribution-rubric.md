# 归因判断规则

用于判断客户 issue support 场景下的责任边界、客观原因、支持路径和 `结论把握度`。

## 判断顺序

1. 问题提出质疑: 反馈是否证明当前真的有问题，还是只能证明有人提出了疑问？
2. 现场还原: 已知事实、时间线、关键缺口和暂定假设是什么？
3. 合同 / 证据: 哪个 POR、SOP、Email、QC、Policy、handoff、release checklist、waiver 或其他证据适用？
4. 交付问题成立性: alleged issue 是否偏离已定义预期，并造成实质交付影响？
5. 破局 / 支持策略: 当前需要冲突扩大、底层溯源、归因防御，还是困境转化，才能形成双赢 support 路径？
6. 履约情况: 谁遵守或没有遵守该约定？
7. 因果关系: 哪个事件或缺失输入实质造成了影响？
8. 归因与 RACI: root-cause owner、fix owner、prevention owner、customer-support owner 是否需要分开？
9. 结论把握度和对外方案: 结论强度有多高，能对客户说什么、做什么？

不要跳过合同 / 证据审查再直接判断 defect。很多争议不是交付问题，而是预期未定义、范围变化、验收标准缺失或证据不足。也不要为了 customer support 接受证据不支持的责任。

## 合同 / 证据测试

只有同时满足以下条件，才使用 `违反约定`:

- 事件发生前已经存在 POR、SOP、Email、QC、Policy、handoff、release checklist、waiver 或其他明确约定；
- 责任方能看到并理解该约定；
- 该约定覆盖争议条件；
- 该方的行为或遗漏违反了该约定；
- 该违反行为实质造成了交付或 support 影响。

当预期未定义、含糊、互相冲突或执行后才变化时，使用 `约定缺口`。

即使结果让某方不满意，只要某方遵守了已定义约定，就归类为 `已遵守约定`，同时给出可对外支持的下一步。

## 交付问题成立性测试

出现以下情况时，归类为 `交付问题尚未成立`:

- 应然状态缺失或含糊；
- 实际行为符合 POR、SOP、Policy、waiver、known limitation 或已记录行为；
- 被报告的影响不在约定交付范围内；
- 严重程度未达到 issue 门槛；
- 没有复现，也没有直接证据；
- 更适合归类为 enhancement、preference 或 late scope change；
- 结论依赖过期证据。

只有存在明确的应然 / 实际差距，且有实质影响时，才归类为 `交付问题成立`。

## 破局 / 支持策略判断

| 策略 | 可采用条件 | 必要证据 / 上下文 | 对客户 support 的目标 | 不能这么用 |
| --- | --- | --- | --- | --- |
| `冲突扩大法` | 单点 issue 被用来推出局部责任，但按该逻辑应影响更多系统范围。 | 系统架构、相关 project、历史 / 当前 release、上下游依赖、提出方判断逻辑原文。 | 把讨论从单点责任拉回一致性检查和共同证据。 | 不能编造牵连范围；不能把“没人投诉”当成唯一证明；不能暗示客户无理。 |
| `底层溯源法` | 争议卡在表层现象或判错基准。 | POR、SOP、Email、QC、Policy、contract、baseline、acceptance criteria、系统 invariant。 | 找到正确判断基准，让客户得到可靠答复。 | 不能无限追问；下一层必须有可验证依据。 |
| `归因防御策略` | 成熟稳定系统突然被归因，且存在外部新变量可能。 | 我方稳定性证据、最近变更记录、监控/回归、上游输入、环境/配置/release 变化。 | 避免无证据承担不该承担的责任，同时定位可行动原因。 | 不能用稳定历史替代证据；不能无脑甩锅。 |
| `困境转化法` | corner / edge / special case 超出单点处理边界。 | 触发条件、Owner/RACI、现有合同覆盖范围、修复和预防 gate。 | 形成 support now、owner clarify、prevent recurrence 的双赢路径。 | 不能把普通本地 bug 包装成系统问题。 |

破局策略只决定支持路径和沟通路径，不直接决定责任归属。责任仍必须回到合同、履约、因果和证据链。

## 问题提出质疑

出现以下情况时，归类为 `问题尚未成立`:

- 报告无法复现，或只基于单一且不清晰的观察；
- 旧 release、旧 upstream 或复制来源项目已有同样行为，且当前合同没有要求改变；
- 相关 project 也有同样行为，且没人说明当前项目为什么必须不同；
- 提出方拿后来的新期待、偏好或范围变化做比较，而不是拿已确认 baseline 做比较；
- 缺少当前证据、影响证据或已确认的评价标准。

事实差异存在但合同含义未证明时，归类为 `只能证明存在差异`。

## 证据强度

| 强度 | 判断标准 | 结论把握度上限 |
| --- | --- | --- |
| `直接证据` | 一手记录能证明事实，例如 ticket、log、commit、测试结果、时间戳、已接受 spec、书面决定 | 高 |
| `相互印证` | 多个独立信号方向一致，但没有单一一手记录证明完整链条 | 中到高 |
| `背景证据` | 只能支持合理性，不能单独证明原因 | 中 |
| `弱证据` | 记忆、转述、含糊观察、单一非一手信号 | 低 |
| `缺失` | 必要证据不存在或未取得 | 暂不能判断 |

不得用弱证据或缺失证据得出高把握度结论。

## 归因类型

`我方责任`
: 我方有明确定义的责任，但没有履行、漏掉必要步骤，或引入了问题。

`其他 Owner 责任`
: 另一个人、团队或系统 Owner 有明确定义的责任，但没有履行。

`上游依赖`
: 必需的上游输入、服务、数据、权限、时间点或决定缺失、错误、延迟或发生变化。

`流程 / 约定缺口`
: 流程本身留下了空白，例如没有 Owner、没有验收标准、没有 review gate、没有升级路径或指令互相冲突。

`环境 / 外部因素`
: 问题实质依赖 runtime、网络、第三方服务、平台策略、市场、硬件或其他不可控条件。

`需求方 / 范围漂移`
: 需求方后续使用了执行前没有确认过的要求来评价结果。

`证据缺口`
: 现有证据不足以安全归因。

`客户支持动作`
: 当前可以对客户或 stakeholder 提供的行动，例如补证据、复现、workaround、clarification、cross-owner review、release gate 修补或正式 reply。

## 结论把握度

- `高`: 有直接证据、清晰约定和明确因果链。
- `中`: 有较好证据，但合同边界或因果细节仍缺一部分。
- `低`: 归因方向合理，但证据弱或不完整。
- `暂不能判断`: 继续归因会变成猜测。

## 红线

- 不推断动机。
- 不把“大家都知道”当成证据。
- 除非约定明确写了沉默等于同意，否则不要把沉默当成批准。
- 不把从未要求某个 Owner 提供的信息缺失归责给该 Owner。
- 如果证据指向我方责任，不隐藏；改用更稳妥的沟通方式。
- 不把 `归因防御策略` 变成默认否认我方责任；一旦监控、回归、最近变更或输入输出证据指向我方，必须降低防御强度。
- 不把 customer support 理解为无条件承认责任；support 的底线是事实、合同、证据和可执行解决路径。
