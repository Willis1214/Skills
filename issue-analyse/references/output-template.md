# 输出模板

默认使用中文字段。`Owner`、`RACI`、`baseline`、`waiver`、`special handle`、`scope drift`、`release gate` 等词可以作为 technical terms 保留；`证据`、`待补证据`、`结论把握度`、`合同边界`、`归因类型` 等默认用中文。

## 问题提出尚未被质疑清楚时

```markdown
## Issue Analyse｜问题提出质疑

### 1. 被提出的问题
| 字段 | 判断 |
| --- | --- |
| 谁提出 | <客户 / 上游 / 下游 / 内部 reviewer / 未知> |
| 他们看到什么 | <原始观察，避免先写成 defect> |
| 他们拿什么对比 | <合同 / spec / 旧版本 / 当前 upstream / 相关 project / 主观期待 / 未知> |
| 当前状态 | 真实存在 / 只证明观察存在 / 尚未证明 / 已排除 |

### 2. 质疑点
| 质疑角度 | 当前判断 | 待补证据 |
| --- | --- | --- |
| 过去有没有 | <旧版本 / 旧项目 / 历史模板是否同样存在> | <旧 release、旧 upstream、历史验收记录> |
| 现在有没有 | <当前 release / 当前 upstream / 当前环境是否仍存在> | <当前包、截图、复现步骤、diff> |
| 相关 project 有没有 | <fully copy 来源项目、兄弟项目、同版本产品是否同样存在> | <相关项目对比结果> |
| 是否有合同基准 | <是否能对应到 PRD / SOP / ticket / checklist> | <合同、验收标准、交付说明> |
| 是否有实质影响 | <是否影响已承诺交付或客户使用> | <影响范围、复现路径、客户场景> |

### 3. 暂定判断
| 字段 | 判断 |
| --- | --- |
| 问题成立性 | 问题成立 / 问题尚未成立 / 只能证明存在差异 / 更像变更请求 / 需要继续核实 |
| 结论把握度 | 高 / 中 / 低 / 暂不能判断 |
| 理由 | <为什么现在只能到这个结论> |

### 4. 下一步最小追问
- <只问能推动“问题是否成立”的最小问题>
```

## 首次信息不足时

```markdown
## Issue Analyse｜现场还原

### 1. 已知事实
| 事实 | 来源 | 稳定性 |
| --- | --- | --- |
| <已确认或用户已描述的事实> | <证据 / 口述 / 待核实> | 直接 / 间接 / 待验证 |

### 2. 关键缺口
| 缺口 | 为什么重要 | 不补会影响什么判断 |
| --- | --- | --- |
| <缺少的事实或材料> | <影响问题成立、合同边界、归因哪一层> | <会导致结论把握度降低或无法归因> |

### 3. 待补证据
| 证据 | 优先级 | 用途 |
| --- | --- | --- |
| <ticket / mail / chat / log / commit / release package / checklist / screenshot> | 高 / 中 / 低 | <用于还原现场、确认应然状态、确认实际状态、确认影响、确认 Owner> |

### 4. 暂定假设
| 假设 | 当前依据 | 状态 |
| --- | --- | --- |
| <可能解释> | <已有证据或背景> | 待验证 / 较可能 / 已排除 / 证据不足 |

### 5. 下一步最小追问
- <只问能推动下一判断节点的最小问题>
```

## 信息足够时

```markdown
## Issue Analyse｜问题归因分析

### 1. 问题提出质疑
| 字段 | 判断 |
| --- | --- |
| 被提出的问题 | <用户描述的问题> |
| 提出方依据 | <合同 / spec / 旧版本 / 当前 upstream / 相关 project / 主观期待 / 未知> |
| 过去是否存在 | 是 / 否 / 未知；<旧版本、旧项目、历史模板情况> |
| 现在是否存在 | 是 / 否 / 未知；<当前 release、当前 upstream、当前环境情况> |
| 相关 project 是否存在 | 是 / 否 / 未知；<fully copy 来源项目、兄弟项目、参考版本情况> |
| 质疑结论 | 问题成立 / 问题尚未成立 / 只能证明存在差异 / 更像变更请求 / 需要继续核实 |
| 结论把握度 | 高 / 中 / 低 / 暂不能判断 |

### 2. 问题是否构成交付问题
| 字段 | 判断 |
| --- | --- |
| 应然状态 | <应当是什么；未知写“未知”> |
| 实际状态 | <实际上是什么；未知写“未知”> |
| 实质影响 | <影响；不足写“未证明达到 issue 级别”> |
| 初步结论 | 问题成立 / 问题尚未成立 / 需要继续核实 |
| 理由 | <为什么成立、不成立或证据不足> |

### 3. 合同与边界
| 合同 / 约定项 | Owner | 证据 | 是否符合 | 缺口 |
| --- | --- | --- | --- | --- |
| <PRD / SOP / API contract / SLA / ticket / handoff agreement / 口头或书面约定> | <Owner> | <证据> | 已符合 / 已违反 / 未定义 / 需要继续核实 | <缺口> |

### 4. 证据链
| 证据 | 来源 | 时间 | 支持的判断 | 强度 | 缺口 |
| --- | --- | --- | --- | --- | --- |
| <证据> | <来源> | <时间> | <支持哪个判断> | 直接证据 / 相互印证 / 背景证据 / 弱证据 / 缺失 | <缺口> |

### 5. 归因矩阵
| 原因 | 归因类型 | Owner / RACI | 证据 | 结论把握度 | 说明 |
| --- | --- | --- | --- | --- | --- |
| <原因> | 我方责任 / 其他 Owner 责任 / 上游依赖 / 流程或约定缺口 / 环境或外部因素 / 需求方或范围漂移 / 证据缺口 | Responsible / Accountable / Consulted / Informed | <证据> | 高 / 中 / 低 / 暂不能判断 | <说明> |

### 6. 沟通边界
| 类型 | 表达 |
| --- | --- |
| 可说 | <证据和合同支持的客观表述> |
| 不应说 | <不该说的推断、攻击或无证据归因> |
| 待补证据 | <还需要补什么证据> |
| 稳妥表达 | <更稳妥、可转发的表达> |

### 7. 下一步行动
| 行动 | Owner | 目的 |
| --- | --- | --- |
| <行动> | <Owner> | <补证据/修合同/修流程/修问题> |
```

## 简短聊天版

如果用户只需要快速讨论，压缩为：

1. `问题提出质疑`;
2. `现场还原`;
3. `问题是否成立`;
4. `合同与边界`;
5. `最强证据`;
6. `可能归因`;
7. `可说 / 不应说 / 待补证据 / 稳妥表达`.

当 `结论把握度` 低于 `高` 时，不得省略 `待补证据`。
