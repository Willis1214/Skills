# 输出模板

默认使用中文字段。`Owner`、`RACI`、`baseline`、`waiver`、`special handle`、`scope drift`、`release gate`、`POR`、`SOP`、`Email`、`QC`、`Policy` 等词可以作为 technical terms 保留；`客户支持目标`、`证据`、`待补证据`、`结论把握度`、`合同边界`、`交付问题`、`归因类型`、`对外解决方案` 等默认用中文。

## 问题提出尚未被质疑清楚时

```markdown
## Issue Analyse｜问题提出质疑

### 1. 被提出的问题
| 字段 | 判断 |
| --- | --- |
| 谁提出 | <客户 / 上游 / 下游 / 内部 reviewer / 未知> |
| 他们看到什么 | <原始观察，避免先写成 defect> |
| 他们拿什么对比 | <POR / SOP / Email / QC / Policy / 旧版本 / 当前 upstream / 相关 project / 主观期待 / 未知> |
| 客户支持目标 | <先补证据 / 先复现 / 先解释边界 / 先给 workaround / 先拉通 Owner> |
| 当前状态 | 真实存在 / 只证明观察存在 / 尚未证明 / 已排除 |

### 2. 质疑点
| 质疑角度 | 当前判断 | 待补证据 |
| --- | --- | --- |
| 过去有没有 | <旧版本 / 旧项目 / 历史模板是否同样存在> | <旧 release、旧 upstream、历史验收记录> |
| 现在有没有 | <当前 release / 当前 upstream / 当前环境是否仍存在> | <当前包、截图、复现步骤、diff> |
| 相关 project 有没有 | <fully copy 来源项目、兄弟项目、同版本产品是否同样存在> | <相关项目对比结果> |
| 是否有合同 / 证据基准 | <是否能对应到 POR / SOP / Email / QC / Policy / handoff / release checklist / waiver> | <合同、验收标准、交付说明、签核记录> |
| 是否有实质影响 | <是否影响已承诺交付、客户使用、验收、安全或合规> | <影响范围、复现路径、客户场景> |

### 3. 暂定判断
| 字段 | 判断 |
| --- | --- |
| 问题成立性 | 问题成立 / 问题尚未成立 / 只能证明存在差异 / 更像变更请求 / 需要继续核实 |
| 结论把握度 | 高 / 中 / 低 / 暂不能判断 |
| 理由 | <为什么现在只能到这个结论> |

### 4. 下一步最小追问
- <只问能推动“问题是否成立”的最小问题>
```

## 首次信息不足时

```markdown
## Issue Analyse｜现场还原

### 1. 已知事实
| 事实 | 来源 | 稳定性 |
| --- | --- | --- |
| <已确认或用户已描述的事实> | <证据 / 口述 / 待核实> | 直接 / 间接 / 待验证 |

### 2. 时间线
| 时间点 | 事件 | Owner / 相关方 | 证据 |
| --- | --- | --- | --- |
| <request / handoff / change / release / complaint / mitigation> | <发生了什么> | <Owner> | <ticket / Email / QC / release record / screenshot> |

### 3. 关键缺口
| 缺口 | 为什么重要 | 不补会影响什么判断 |
| --- | --- | --- |
| <缺少的事实或材料> | <影响合同边界、交付问题、归因或对外方案哪一层> | <会导致结论把握度降低或无法归因> |

### 4. 待补证据
| 证据 | 优先级 | 用途 |
| --- | --- | --- |
| <POR / SOP / Email / QC / Policy / handoff / release checklist / waiver / ticket / log / screenshot> | 高 / 中 / 低 | <用于还原现场、确认应然状态、确认实际状态、确认影响、确认 Owner> |

### 5. 暂定假设
| 假设 | 当前依据 | 状态 |
| --- | --- | --- |
| <true defect / undefined expectation / legacy latent issue / waiver / upstream drift / scope drift / evidence gap> | <已有证据或背景> | 待验证 / 较可能 / 已排除 / 证据不足 |

### 6. 下一步最小追问
- <只问能推动下一判断节点的最小问题>
```

## 信息足够时

```markdown
## Issue Analyse｜客户 Issue Support 分析

### 1. 问题提出质疑
| 字段 | 判断 |
| --- | --- |
| 被提出的问题 | <用户描述的问题，避免先写成我方 defect> |
| 提出方依据 | <POR / SOP / Email / QC / Policy / 旧版本 / 当前 upstream / 相关 project / 主观期待 / 未知> |
| 过去是否存在 | 是 / 否 / 未知；<旧版本、旧项目、历史模板情况> |
| 现在是否存在 | 是 / 否 / 未知；<当前 release、当前 upstream、当前环境情况> |
| 相关 project 是否存在 | 是 / 否 / 未知；<fully copy 来源项目、兄弟项目、参考版本情况> |
| 质疑结论 | 问题成立 / 问题尚未成立 / 只能证明存在差异 / 更像变更请求 / 需要继续核实 |
| 客户支持目标 | <先解释 / 先复现 / 先补证据 / 先 workaround / 先 review / 先修复> |
| 结论把握度 | 高 / 中 / 低 / 暂不能判断 |

### 2. 现场还原
| 项目 | 判断 |
| --- | --- |
| 已知事实 | <事实链> |
| 时间线 | <request -> handoff -> change -> release -> complaint -> mitigation> |
| 关键缺口 | <缺少什么> |
| 暂定假设 | <多个可能解释及状态> |

### 3. 合同与证据
| 合同 / 证据项 | Owner | 证据 | 是否符合 | 缺口 |
| --- | --- | --- | --- | --- |
| <POR / SOP / Email / QC / Policy / handoff / release checklist / waiver / known limitation> | <Owner> | <证据> | 已符合 / 已违反 / 未定义 / 需要继续核实 | <缺口> |

### 4. 交付问题判断
| 字段 | 判断 |
| --- | --- |
| 应然状态 | <来自 POR / SOP / Email / QC / Policy / baseline / waiver；未知写“未知”> |
| 实际状态 | <当前证据证明的实际状态；未知写“未知”> |
| 实质影响 | <影响；不足写“未证明达到交付 issue 级别”> |
| 初步结论 | 交付问题成立 / 交付问题尚未成立 / 只能证明存在差异 / 需要继续核实 |
| 理由 | <为什么成立、不成立或证据不足> |

### 5. 破局 / 支持策略
| 方法 | 当前是否采用 | 推演 / 追问 / 防御点 | 对客户 support 的目标 | 风险边界 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| 冲突扩大法 | 是 / 否 / 待定 | <如果该逻辑成立，应牵连哪些模块、release、相关 project 或上下游> | <一致性检查 / 共同补证> | <只推导有系统依据的范围> | <要核对的对比对象> |
| 底层溯源法 | 是 / 否 / 待定 | <需要追到的 POR / SOP / Email / QC / Policy / baseline / invariant> | <确认正确判断基准> | <下一层必须可验证> | <要补的基准证据> |
| 归因防御策略 | 是 / 否 / 待定 | <成熟系统稳定性证据 + 外部新变量假设> | <避免无证据承担责任，同时找到可行动原因> | <不能用稳定历史替代证据> | <要核对的变更、输入、监控或回归> |
| 困境转化法 | 是 / 否 / 待定 | <为什么是 corner / edge / special 或系统缺口> | <support now / owner clarify / prevent recurrence> | <说明为何单点行动不足> | <需要拉通的 Owner 和 review 议题> |

### 6. 归因矩阵
| 原因 | 归因类型 | Owner / RACI | 证据 | 结论把握度 | 说明 |
| --- | --- | --- | --- | --- | --- |
| <原因> | 我方责任 / 其他 Owner 责任 / 上游依赖 / 流程或约定缺口 / 环境或外部因素 / 需求方或范围漂移 / 证据缺口 | Responsible / Accountable / Consulted / Informed | <证据> | 高 / 中 / 低 / 暂不能判断 | <说明> |

### 7. 沟通边界与对外方案
| 类型 | 表达 |
| --- | --- |
| 可说 | <证据和合同支持的客观表述> |
| 不应说 | <不该说的推断、攻击或无证据归因> |

### 8. 下一步行动
| 行动 | Owner | 目的 |
| --- | --- | --- |
| <行动> | <Owner> | <补证据 / 复现 / workaround / 修合同 / 修流程 / 修问题 / 对外回复> |
```

## 简短聊天版

如果用户只需要快速讨论，压缩为：

1. `问题提出质疑`;
2. `现场还原`;
3. `合同与证据`;
4. `交付问题判断`;
5. `破局 / 支持策略`;
6. `归因矩阵`;
7. `可说 / 不应说 / 待补证据 / 稳妥表达 / 对外解决方案 / 双赢路径`.

当 `结论把握度` 低于 `高` 时，不得省略 `待补证据`。当面向客户 support 时，不得省略 `对外解决方案`。
